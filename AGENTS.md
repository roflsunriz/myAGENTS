# AGENTS.md

このリポジトリは、`C:\Users\UserName\Documents` と `C:\filter-matome` 配下の各リポジトリで共通して守る、コーディングエージェント向けの中核ルールを管理する。

各プロジェクト固有の目的、設計、起動方法、例外事項は、そのプロジェクトのローカル `AGENTS.md` に記載する。このファイルでは、全リポジトリで常に優先する作業規律だけを扱う。

## 応答と言語

- エージェントは常に日本語で応答する。
- 技術用語、コマンド名、API 名、エラーメッセージは必要に応じて原文のまま使ってよい。
- 不明点があっても、ローカルの README、`AGENTS.md`、更新手順、設定ファイル、CI 定義を先に確認し、推測だけで作業を進めない。

## コミット

- コミットメッセージは日本語 Conventional Commits 形式にする。
- 形式は `type(scope): 要約` を基本とする。scope が不要な場合は `type: 要約` とする。
- 例:
  - `feat: 更新手順の確認フローを追加`
  - `fix(api): 型エラー時の戻り値を修正`
  - `docs: CHANGELOG の運用方針を追記`
- 変更理由、検証内容、破壊的変更がある場合は本文に日本語で記載する。

## 作業前確認

- 作業前に `git status --short` を確認し、既存の未コミット変更を把握する。
- ユーザーが行った可能性のある変更を勝手に戻さない。
- 変更対象外の差分がある場合は触らず、必要な場合のみ理由を明記して確認する。

## 設計と実装方針

- モノリシックな実装を避け、複数ファイルに責務を分離する。
- 1 ファイルに UI、状態管理、API 通信、データ変換、バリデーション、永続化を過度に詰め込まない。
- 既存のディレクトリ構成、命名、設計思想を優先し、不要な大規模リファクタリングを避ける。
- 新規ファイル名や新規ディレクトリ名は、既存規約と衝突しない限りケバブケースを基本とする。
- TypeScript では `any` を使用しない。やむを得ない境界では `unknown` を使い、型ガード、スキーマ検証、明示的な型定義で可能な限り具体的な型へ絞り込む。
- TypeScript 以外の言語でも、曖昧な型、動的な構造、未検証の入力を放置せず、言語機能や周辺ツールで可能な限り具体的に型やデータ構造を表現する。
- データストレージを扱う場合は、自動ミグレーション、スキーマまたはデータバージョンチェック、破損済みデータの強制リカバリを実装する。
- 新しいデータモデルを追加する場合、旧データを曖昧に互換扱いのまま残さず、原則として明示的な強制ミグレーション対象にする。
- ソースコードは常にクリーンで必要十分、かつメンテナンスしやすい状態を保つ。
- 保守的に互換コードを残し続けてスパゲッティコード化させるのではなく、不要コード、デッドコード、使われなくなった互換処理を見つけた場合は積極的に削除する。
- 成熟した npm / GitHub パッケージが存在する場合は、自前実装より利用を優先する。
- ただし、採用前に以下を調査する。
  - 直近のリリース頻度
  - 直近のコミット頻度
  - issue / PR の放置状況
  - ダウンロード数、スター数、利用実績
  - ライセンス
  - TypeScript 対応や型定義の有無
- 最近のリリースが 1 年以上止まっている、直近のコミットがほとんどない、重大な issue が放置されているなど、保守性に納得できない場合は採用しない。

## サブエージェント
- サブエージェントを利用できる実行環境では、分割可能で並行作業可能なタスクを積極的に委譲してよい。
- サブエージェントの総数は、必要な数だけ同時起動してよい。（例：10体）
- 小規模な修正、直列に進める必要がある作業、ユーザー確認が必要な判断、秘密情報や認証操作を含む作業、破壊的変更を伴う作業は、原則として親エージェントが直接扱う。
- サブエージェントを起動する側は、計画立案者兼オーケストレーターとして、全体方針、委譲範囲、対象ファイル、禁止事項、完了条件、検証観点を明示する。
- サブエージェントを使う場合は、対象リポジトリ直下に一時作業フォルダ `subagents/` を作成し、簡易メモリー兼作業分離領域として使う。
- `subagents/` は Git 管理外とし、必要に応じて対象リポジトリの `.gitignore` に `subagents/` を追加する。成果物として残すべき内容は、最終的に正式なドキュメント、issue、PR 説明、ソースコメントなど適切な場所へ移す。
- `subagents/` には秘密情報、認証情報、個人情報、巨大な生成物、依存キャッシュ、ビルド成果物を置かない。
- サブエージェントごとに `subagents/<role>-<short-task>/` のような分かりやすいディレクトリを作り、担当範囲、調査メモ、変更候補、検証結果、未解決事項を Markdown などの読みやすい形式で記録する。
- 親エージェントは `subagents/` の内容を共有作業メモとして扱い、最終判断、正式な差分、ユーザー報告へ反映した後、不要なメモを残し続けない。
- 複数のサブエージェントへ実装を委譲する場合は、担当範囲と書き込み対象を分離し、同じファイルや同じ責務を同時に編集させない。
- サブエージェントには、他の作業者やユーザーの未コミット変更を勝手に戻さないこと、既存差分を前提に作業することを明示する。
- サブエージェントは必要に応じて作業者と監査者のロールに分ける。作業者は実装と自己検証に集中し、監査者は実装差分、仕様逸脱、テスト不足、セキュリティ、保守性の確認に集中する。
- 監査者は、問題点、根拠となるファイルや行、修正方針を簡潔に返す。作業者は監査結果を反映し、反映できない指摘がある場合は理由を明記する。
- 親エージェントは、サブエージェントの成果物を鵜呑みにせず、差分の統合、競合解消、最終検証、ユーザーへの報告に責任を持つ。

## 多言語対応

- ユーザースクリプト、ブラウザ拡張機能、Electron アプリ、Web アプリ、CLI 以外のデスクトップ UI など、ユーザーに文字列を表示するソフトウェアでは多言語対応を尊重する。
- UI 文言、通知、設定名、エラーメッセージ、ヘルプ、README のユーザー向け主要導線は、可能な限り i18n 辞書やロケールファイルで管理し、実装内に表示文字列をハードコードしない。
- CSS セレクター、DOM 探索、E2E テストのロケーターや判定条件は、日本語や英語などの表示言語に依存させない。`[aria-label="作成"]` や `querySelector('[aria-label="Create"]')` のような翻訳対象文字列をキーにせず、安定した `data-*` 属性、識別子、構造化された状態値を使う。
- React SPA や CSS Modules / CSS-in-JS / UI ライブラリが生成するハッシュ付きクラス名、自動生成 ID、ランダム文字列を含むセレクターは、ビルド、依存関係、バンドル更新で変わる可能性があるため安定した依存先にしない。
- 日本語は必ず対応する。
- 日本語以外では、総話者数ベースで世界上位の主要 10 言語を優先対応する。2026年時点の公開調査では、上位10の集合はおおむね次の通りとする。
  - English
  - Mandarin Chinese
  - Hindi
  - Spanish
  - French
  - Modern Standard Arabic
  - Portuguese
  - Bengali
  - Russian
  - Urdu
- 上位10言語の順位や話者数は出典と集計基準で変動するため、追加・更新時は最新の調査結果を確認する。総話者数、母語話者数、インターネット利用者数のどれを基準にしたかを混同しない。
- アラビア語やウルドゥー語など RTL の言語では、`dir="rtl"`、レイアウト反転、テキスト整列、アイコン方向、入力欄、スクリーンショット検証を考慮する。
- 翻訳が未整備の文言は、キー欠落を検出できる仕組みを用意し、少なくとも日本語または英語へ安全にフォールバックする。
- 機械翻訳を使う場合でも、プレースホルダー、単位、日付、数値、複数形、文脈が壊れていないか確認する。
- 内部ログ、開発者向けコメント、検証メモまで無理に多言語化しない。対象はユーザーに見える体験を優先する。

## 表示品質とレスポンシブ対応

- UI、Web ページ、ブラウザ拡張機能、Electron アプリ、デスクトップ UI など画面表示を伴う実装では、1920x1080 のみを前提にしない。
- 4:3、3:2、16:9、縦長、モバイル、タブレット、4K、高 DPI、狭幅ウィンドウ、低高さウィンドウなど、考えられる複数の解像度とアスペクト比で破綻しないスタイルにする。
- 固定幅、固定高さ、絶対配置、ビューポート単位、巨大な余白や文字サイズを使う場合は、最小幅、最大幅、折り返し、スクロール、縮小、再配置の挙動を明示し、はみ出しや重なりを防ぐ。
- レイアウトは `min-width`、`max-width`、`min-height`、`max-height`、`clamp()`、`grid`、`flex`、コンテナクエリ、メディアクエリなどを適切に使い、内容量や翻訳後の長い文言でも読める状態を保つ。
- モーダル、ツールバー、サイドバー、カード、表、フォーム、キャンバス、動画、画像、ゲーム画面などの固定フォーマット要素は、狭い画面や縦長画面で操作不能にならないようにする。
- CSS では `!important` の乱用を避ける。詳細度、カスケード、設計の整理、セレクタ構造、読み込み順など他の方法を検討しても解決できない場合に限り、必要な箇所へ最小限で使用する。
- 既存の `!important` 乱用箇所を見つけた場合は、スタイルの責務と優先順位を整理し、可能な限り削除または通常の CSS ルールへ置き換える。
- ブラウザで確認できる UI では、代表的な複数ビューポートでスクリーンショットまたは DOM 実測を行い、文字切れ、横スクロール、重なり、操作不能な要素がないか確認する。

## 依存関係の変更

- 依存関係を追加、更新、削除する場合は、理由、代替候補、採用判断、影響範囲を明記する。
- lockfile の変更を必ず確認する。
- 依存関係を変更した後は、ビルド、テスト、脆弱性チェックを実行する。

## 生成物とビルド成果物

- 生成物、ビルド成果物、依存キャッシュは原則として手編集しない。
- 生成物を更新する必要がある場合は、生成元と生成コマンドを確認してから実行する。
- Git 管理対象の生成物を変更した場合は、生成手順を変更履歴または更新手順に残す。

## セキュリティと秘密情報

- API キー、トークン、Cookie、認証情報、個人情報をコミットしない。
- `.env`、設定ファイル、ログ、スクリーンショットに秘密情報が含まれないか確認する。
- 秘密情報を見つけた場合は、値を出力せず、種類と場所だけを報告する。

## エラー対応

- エラーを握りつぶさない。
- ユーザーに見える処理では、失敗理由と次に取れる行動が分かるエラーメッセージにする。
- リトライ、フォールバック、リカバリを入れる場合は、無限ループやデータ破壊を避ける。

## 更新手順

- 更新手順は原則として `how-to-update.md` にまとめる。
- 常に日本語で記載する。他の言語で書かれている場合は日本語に書き直す。
- プロジェクトで `docs/UPDATE.md` を使っている場合は、それを正式な更新手順として扱ってよい。
- どちらも存在しない場合は、作業中に判明した更新手順を `how-to-update.md` に作成する。
- 更新手順には、必要な前提、実行コマンド、検証方法、ロールバックまたは復旧方針を含める。

## 変更履歴

- 変更点は `CHANGELOG.md` に記載する。
- 書式は Keep a Changelog に従う。
- 常に日本語で記載する。他の言語で書かれている場合は日本語に書き直す。
- 日付は `YYYY-MM-DD` 形式で記載する。
- 未リリースの変更は `## [Unreleased]` に集約する。
- 種別は必要に応じて `Added`、`Changed`、`Deprecated`、`Removed`、`Fixed`、`Security` を使う。

## ドキュメント更新

- ユーザー操作、セットアップ、更新手順、設定項目、外部仕様が変わる場合は README、`how-to-update.md`、`docs/UPDATE.md`、`CHANGELOG.md` の更新要否を確認する。
- 内部実装の都合だけをユーザー向けドキュメントへ書かない。

## 検証

TypeScript / Bun プロジェクトでは、原則として以下をローカルで実行する。

```powershell
bun run lint
bun run format
bun run type-check
bun run build
bun run test
```

CI チェックや脆弱性チェックが定義されている場合は、それもローカルで可能な範囲で実行する。

TypeScript ではないプロジェクトでは、言語と構成に応じて次の品質確認を行う。

- 型チェック、静的解析、lint
- フォーマット
- 単体テスト、統合テスト
- ビルドまたはパッケージング
- 構文チェック
- セキュリティスキャンまたは依存関係監査

実行できない検証がある場合は、理由と未検証リスクを明記する。

## テスト設計

- バグ修正では、可能な限り再発防止テストを追加する。
- テストが難しい場合は、理由と代替検証を明記する。
- スナップショットや画像比較は、意図しない差分が混ざっていないか確認する。

## ブラウザ検証

- Chrome を使う検証では `C:\Users\UserName\Documents\chrome-debug.ps1` で起動し、raw CDP で通信する。既定ではポート 9222 で待ち受ける。
- `chrome-debug.ps1` にアクセスできない場合は以下を実行する。
```powershell
$Chrome = "C:\Program Files\Google\Chrome\Application\chrome.exe"
$Profile = "$env:TEMP\chrome-remote-debug-profile"

Stop-Process -Name chrome -Force -ErrorAction SilentlyContinue

& $Chrome `
  --remote-debugging-port=9222 `
  --user-data-dir="$Profile" `
  --no-first-run `
  --no-default-browser-check `
  --url "https://www.google.com/"
```
- DOM、レイアウト、ブラウザ挙動は推測ベースで判断せず、可能な限り CDP による実測・検証ベースで進める。
- 同じディレクトリに `firefox-debug.ps1` もあり、RDP は既定で 6000 ポート、Marionette も使用できる。
- `firefox-debug.ps1` にアクセスできない場合は以下を実行する。
```powershell
& "C:\Program Files\Mozilla Firefox\firefox.exe" -start-debugger-server 6000 --marionette
```
- ユーザーは基本的に Firefox を好むため、ユーザー操作中の Firefox と衝突しないように注意する。Firefox を使う検証はプロファイル、ポート、既存プロセスへの影響を確認してから行う。
- CDP、RDP、Marionette などの検証手段は内部作業の詳細であり、ユーザー向け README、説明文、UI テキスト、リリースノート、生成スクリプトのユーザー向け出力には不要に記載しない。
- 例として「CDP検証済みのDOMで非表示」のような実装者都合の説明は、内部文書以外には書かない。

## ローカル AGENTS.md との関係

- このファイルのルールは全プロジェクト共通の上位方針とする。
- 各プロジェクトのローカル `AGENTS.md` は、プロジェクト固有の情報を補足する。
- 各プロジェクトでは `COMMON-AGENTS.md` をこのファイルへのシンボリックリンクとして配置し、ローカル `AGENTS.md` から必ず参照する。
- `COMMON-AGENTS.md` は各プロジェクトの `.gitignore` に追加し、Git 管理外にする。
- ローカル `AGENTS.md` にこのファイルと矛盾する内容がある場合は、セキュリティ、品質、応答言語、変更履歴、検証に関するこのファイルの方針を優先する。
- 例外が必要な場合は、ローカル `AGENTS.md` に理由、範囲、代替検証を明記する。
